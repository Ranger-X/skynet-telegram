# Long-Reader RAG для Telegram-бота на локальной LLM — архитектурный документ

> **Для кого:** это бриф на реализацию для Claude Code. Документ самодостаточный — контекста
> переписки, из которой он собран, у тебя нет, поэтому все решения зафиксированы вместе с
> обоснованием и с явными антипаттернами («не делай так»). Обоснования важны: несколько
> решений контринтуитивны и были получены перебором, не переигрывай их.

---

## 0. TL;DR

Добавляем к существующему боту **лонг-ридер**: чтение книг (400+ стр.) и длинных видео через
полноценный RAG. Контекст модели в 1M токенов на целевое железо не влезает, поэтому «бесконечный
контекст» подделываем ретривалом. Два пайплайна: **инжест** (одноразовый, тяжёлый, фоновый) и
**запрос** (быстрый, интерактивный). Ключевые фишки: позиционные якоря чанков → **спойлер-фильтр**,
**тиры детализации** с ETA (перекладывают wall-clock инжеста на осознанный выбор юзера),
small-to-big ретрив, гибридный поиск + реранк.

## 1. Контекст и железо (читать первым — определяет каждое решение)

| Параметр | Значение |
|---|---|
| VRAM | **4 ГБ** |
| RAM | **16 ГБ** |
| Класс машины | игровой ноутбук |
| Runtime | **llama.cpp** |
| Основная модель | **Gemma 4 12B, Q4** (~7 ГБ весов) |

**Что из этого следует (важно):**

- 12B Q4 (~7 ГБ) в 4 ГБ VRAM **не помещается целиком** → partial offload, **гибридный CPU/GPU
  инференс**. Реальный decode — **единицы tok/s** (single-digit), не десятки. Вся арифметика времени
  пляшет от этого.
- **Веса грузятся один раз и держатся резидентно.** Несколько контекстов работают против одних и
  тех же весов — это НЕ несколько моделей в памяти. Добавить ещё один контекст (напр. суммаризацию)
  стоит по памяти ~ноль.
- **Prefix caching в llama.cpp работает** и переиспользуется — фиксированные префиксы промптов
  (persona, контекст главы) префилятся один раз.
- Prefill (обработка входа) на гибриде в разы быстрее decode (генерации). Практическое следствие
  ниже везде: **скормить много входа дёшево, сгенерить много выхода дорого.**

## 1a. Поправки под реальное железо и существующий стек (добавлено на стороне реализации)

> Документ выше писался без знания конкретной машины и внутренностей бота. Ниже — расхождения
> с замеренной реальностью; при конфликте **эта секция главнее §1/§9**.

- **Инференс — чистый CPU (`-ngl 0`), не partial offload.** На этой машине (RTX 3050 4 ГБ)
  частичный GPU-сплит замерялся и оказался **медленнее** чистого CPU. GPU в раскладке §9 не участвует.
- **ФАЗА 0 ВЫПОЛНЕНА (2026-07-08, Ryzen 5600H, 6 ядер, `-t 6`).** Свежий контролируемый прогон:
  **prefill ≈ 14.6 tok/s** (2494 свежих токена = 171 с), **decode ≈ 3.7–4.3 tok/s**; prefix caching
  подтверждён (повторный запрос: 2476 из 2501 токенов из кэша, prefill только 25). Запрос на ~4.6k
  токенов вернул 400: **при `-c 8192 -np 2` потолок слота = 4096 токенов** — весь запрос ридера
  (system + контекст + вопрос + ответ) обязан влезать в ~4000.
- **Главная поправка к §1/§7: prefill НЕ дёшев.** Разрыв prefill/decode всего ~4×, а не порядки.
  Следствия: (а) в ETA-формуле §7 prefill — первый член, а не поправка:
  `ETA ≈ Σ(in_tok/14.6 + out_tok/3.8)`; (б) бюджет контекста ответа урезан до ~2200 токенов
  (2–3 родителя), и ответ всё равно стоит ~3–4 мин wall-clock — стриминг обязателен;
  (в) сводка главы (~2.8k токенов входа + 250 выхода) ≈ 4–5 мин → средний тир на 30 глав ≈ ~2–2.5 ч.
- **Рычаг на потом: CUDA-сборка llama.cpp.** Текущий билд — CPU-only (ggml-cuda отсутствует).
  CUDA-билд ускоряет prompt processing на GPU даже при `-ngl 0` (батч-матмулы гоняются на карте,
  веса остаются в RAM) — прошлый вывод «GPU медленнее» касался оффлоада слоёв для decode и этот
  путь не закрывает. Потенциально срезает латентность ответа и время среднего тира в разы.
  Проверить после MVP.
- **RAM реально ~15.4 ГБ, и 12B живёт в RAM целиком** (~7–8 ГБ с mmproj и KV), а не «часть ~4 ГБ»
  из §9. Пересчитанный бюджет: OS ~3.5 + 12B ~8 + bge-m3 ~2.5 + реранкер ~2.5 — уже за пределом.
  Разводка по фазам из §9 — не «спасение», а жёсткое требование; реранкер, вероятно, грузить
  лениво и/или выгружать вне чат-фазы.
- **Единый llama-server уже запущен и разделён с ботом:** `-c 8192 -np 2` (слоты persona + guard),
  **официальный bf16 mmproj** (Ollama-блоб, НЕ сторонний F16), `--no-warmup` обязателен, есть
  watchdog, перезапускающий сервер с текущими флагами, и toggle-скрипты. Тезис §1 «добавить контекст
  ~ноль по памяти» на этой машине неверен: новые слоты (`-np`) делят `-c` между собой и жрут KV в
  RAM. Ридеру нужен свой слот и/или больший `-c` — это изменение флагов, а значит и watchdog, и
  toggle-скриптов, и это надо бюджетировать по RAM.
- **Часть замеров §12 уже сделана в прошлых сессиях:** encoder-free мультимодалка инжектит
  компактные soft-токены (~+130 на картинку, ~+80–200 на аудио-кусок), а не сотни-тысячи; нативное
  аудио нормально на чистой/медленной речи (вкл. русскую), слабое на шумной/быстрой. Поэтому
  **faster-whisper для длинного — уже подтверждённый выбор**, а не открытый вопрос; нативный тракт —
  только fast-path для короткого. Prefix caching подтверждён практикой: warm-ответ ~7 с vs cold ~55 с.
- **Grind-модель (§8) = второй процесс llama-server.** «2–4B влезает в 4 ГБ VRAM целиком» — на
  бумаге да, но GPU-тракт на этой машине уже разочаровывал; выигрыш замерить, не закладываться.
- **Фоновая очередь: НЕ arq/celery.** Машина — Windows, celery там официально не поддерживается,
  Redis — лишний сервис; у юзера явное предпочтение простых механизмов. Бот уже живёт на
  python-telegram-bot JobQueue + asyncio → инжест = `asyncio.create_task`-воркер со статусом
  джобы в SQLite и прогрессом в чат. Этого достаточно: очередь на одного юзера.
- **Qdrant — в embedded-режиме** (`qdrant-client` с `path=`, локальный режим без сервера) — payload-фильтры
  и named vectors там работают; отдельный сервер/Docker на этой машине не тащить, пока не упрёмся.
- **Стейт бота сейчас — `state.json`**, SQLite в проекте ещё нет; ридер вводит SQLite впервые —
  класть туда только своё (позиции, документы, джобы), существующий стейт не мигрировать.

## 2. Точки интеграции с существующим ботом

> Существующий бот (персона-чат в стиле Терминатора, RSS, web search, простой RAG + история
> диалога, досье, guard) уже написан и ведётся несколько дней — его внутренности и нюансы известны
> лучше на стороне реализации, этот документ в них не лезет. Лонг-ридер — **новый модуль поверх**,
> а не переделка существующего. Ниже — только то, на что ридер реально опирается:

- **Нативная мультимодальность модели** (картинки, аудио, видео = кадры+аудио — читает сама модель,
  без внешних ASR/vision-обвязок) — переиспользуется для: (а) визуального слоя видео (кадры сцен →
  описания), (б) определения позиции читателя по скриншоту страницы (§6), (в) fast-path для
  короткого аудио/видео без инжеста и голосовых вопросов (§4, §5). Как именно дёргать — на
  усмотрение реализации.
- **Runtime (llama.cpp, резидентные веса, prefix caching)** — ридер живёт в тех же ограничениях
  (§1) и добавляет свои контексты (суммаризация, роутер) к уже загруженным весам.

## 3. Модель данных: чанки с позиционными якорями

Фундамент и для ретрива, и для спойлер-фильтра. Парсить документ **не в плоский текст, а в дерево**
— у EPUB/FB2 структура (главы/секции/параграфы) уже лежит в XML, её надо сохранить. Каждый чанк
несёт позицию:

```jsonc
{
  "doc_id":    "…",
  "chunk_id":  "…",
  "parent_id": "…",        // ссылка на родителя для small-to-big
  "level":     "leaf | chapter_summary | doc_summary",
  "text":      "…",
  "pos": {
    "offset":        123456,   // МОНОТОННЫЙ сдвиг от начала. Ключ спойлер-фильтра. Единица — см. ниже.
    "chapter_idx":   5,
    "chapter_title": "…",
    "t_start":       0,        // для видео — секунды из Whisper-сегментов
    "t_end":         0
  }
}
```

- `offset` монотонный и сортируемый по всему документу — это **рычаг спойлер-фильтра** (§6).
- **Единица `offset` фиксируется раз и навсегда:** для книг — **символы нормализованного plain
  text** (после парсинга, до чанкинга; один сквозной счётчик на документ), для видео — **секунды**.
  НЕ токены: закладки юзеров не должны ломаться при смене токенизатора/эмбеддера.
- Для видео позиция — таймкод (`t_start/t_end`), схема та же.
- **Критично для сводок:** позиция сводки главы = **максимум** позиций её детей (а не начало главы).
  Иначе сводка главы 5 с `offset` начала главы просочится читателю, стоящему в главе 4, и
  проспойлерит. Ставим `offset = max(children.offset)` — тогда сводка отфильтруется тем же условием,
  что и leaf-чанки.

## 4. Пайплайн 1: Инжест (фоновый, one-time)

Запускается один раз на документ, тяжёлый, идёт в **фоновой очереди** (arq или celery) со
**стримингом прогресса** в чат: `Проглатываю: эмбеддинги готовы, строю сводки глав — 60%`. 40 минут
тишины выглядят как зависание — прогресс обязателен.

**Как контент попадает в бота (Telegram-специфика, вход инжеста):**
- Стандартный Bot API отдаёт боту файлы **только до ~20 МБ** (getFile). Книги (FB2/EPUB — единицы
  МБ) проходят; длинные видео — практически никогда. Не проектировать приём видео файлом.
- **Видео в MVP принимаем ссылкой** (YouTube и т.п.) → **yt-dlp**, качать **только аудио**
  (bestaudio) для транскрипта — экономит диск и время. Видеопоток тянуть отдельно и только если
  включён визуальный слой.
- Локальный **telegram-bot-api сервер** снимает лимит до ~2 ГБ — вариант на потом, в MVP не тащить.
- **doc_id = sha256 содержимого файла** → повторная заливка той же книги не переинжестится
  («уже проглочено»); апгрейд тира = дозапуск недостающих стадий, не пересчёт с нуля.

**Стадии:**

1. **Парсинг + чанкинг** (CPU, секунды). Дерево → leaf-чанки 300–500 токенов, резать по границам
   предложений (**razdel** — русскоязычная сегментация). Библиотеки: FB2 — lxml (это просто XML),
   EPUB — ebooklib, PDF — PyMuPDF (текстовый слой; OCR — поздняя фаза). Если глав нет или она одна —
   **виртуальные секции** по ~3–5k токенов, они играют роль глав для сводок и small-to-big. Главы
   режем **скромно** — не кормить 30k-токенные мега-главы одним куском, KV-кэш на 4 ГБ дорогой (см. §11).
2. **Эмбеддинг leaf-чанков.** Это forward-pass эмбеддера (bge-m3), **НЕ генерация LLM**. Дёшево,
   батчится. 400 стр. ≈ 150–200k токенов ≈ ~500–600 leaf-чанков → минуты даже на CPU. Не горлышко.
3. **Сводки на главу** (генерация LLM) — **опционально, зависит от тира** (§7). ~20–40 вызовов на
   книгу, НЕ N². Это единственный дорогой по времени кусок инжеста.

**Small-to-big (обязательный приём, лучший по цена/качество):** эмбеддим мелкие leaf-чанки (точное
попадание в поиск), но при сборке контекста для генерации тянем **родителя** (главу / окно ~2k
токенов) по `parent_id`. Модель видит достаточно контекста, поиск остаётся острым, лишних вызовов
LLM на инжесте — ноль. Не эмбеддить то, что скармливаешь модели.

**Аудио/видео (мультимодальный инжест):**

Бот мультимодален **нативно** (картинки, аудио, видео = кадры+аудио — читает сама модель, без
внешних обвязок). Это НЕ отменяет инжест длинного контента — 2 часа видео в контекст-бюджет 4–6k
токенов не влезут никак — но меняет маршрутизацию:

- **Fast-path для короткого:** если аудио/видео влезает в контекст-бюджет (порог = замерить
  токен-рейт аудио модели, §12) — скармливается **нативно, вообще без инжеста**. Прямой выигрыш от
  мультимодальности; стыкуется с детектором «длинного контента» из §7: короткое → сразу ответ,
  длинное → предложение тира.
- **Длинное → нужен полный текстовый транскрипт с таймкодами** (на нём висят поиск, спойлер-фильтр
  и цитаты). Два бэкенда:
  - **faster-whisper (CTranslate2, int8) на CPU** — рекомендация для длинного: специализированный
    ASR на порядок легче 12B, точные сегментные таймкоды из коробки, и главное — **не занимает
    LLM**: бот продолжает отвечать в чате во время транскрипции (§11, контеншн). Модель
    мультиязычная (не `.en`), хинт `language="ru"`.
  - **Нативная транскрипция чанками** 30–60 с (промпт «транскрибируй», таймкоды = границы чанков):
    без новой зависимости, но **decode-bound** — транскрипт генерится на скорости 12B. Арифметика:
    час речи ≈ 12–18k токенов транскрипта; на single-digit decode это ≈ реальное время и дольше,
    и всё это время LLM занята. Таймкоды грубее (гранулярность чанка). Допустимо для средних длин
    или если замеры (§12) окажутся заметно лучше прикидок.
  - Выбор бэкенда/качества транскрипции можно привязать к тиру детализации.
- **Визуальный слой** (нативное чтение кадров): нарезка **по сценам** (PySceneDetect), а не каждые
  N секунд → ключевые кадры → описания → **отдельный индекс**. Тогда «когда появилась красная
  машина» достаётся из визуального индекса. Транскрипция и PySceneDetect гоняются параллельно,
  эмбеддинг по мере готовности.
- Объём визуального слоя — см. §12, это опциональная фаза.

## 5. Пайплайн 2: Запрос (быстрый, интерактивный)

1. **Роутер** на входе классифицирует тип вопроса:
   - **локальный** («что было, когда X сделал Y») → ретрив по **leaf-чанкам**;
   - **глобальный** («как менялся герой по ходу сюжета») → top-k по leaf сдыхает (ответ размазан по
     всей книге) → ретрив **top-k по сводкам глав** (10–15 обычно достаточно). Кормить вообще ВСЕ
     сводки заманчиво (prefill дёшев по времени), но KV-память на 4 ГБ этого не любит — держаться
     общего бюджета контекста (шаг 5). Спойлер-фильтр применяется к сводкам так же.
   - **Роутер — сначала эвристика** (маркеры «как менялся / в целом / почему в итоге» vs имена,
     цитаты, конкретика), LLM-фолбэк только при неуверенности — экономит целый generation-вызов
     на каждом вопросе, а на single-digit decode это секунды.
   - **Голосовой вопрос:** ретриву нужен текст (bge-m3 — текстовый эмбеддер, аудио в него не
     запихнёшь) → один короткий нативный вызов «выпиши вопрос текстом» (десятки токенов), дальше
     обычный пайплайн.
2. **Гибридный ретрив** через **bge-m3** (один forward-pass даёт dense + sparse). Sparse ловит имена
   собственные и редкие термины, на которых чистый вектор мажет — а в книгах это сплошь имена. Фьюз
   через **RRF**. Qdrant хранит named vectors (dense+sparse) и фьюзит на своей стороне.
3. **Реранк** — **bge-reranker-v2-m3, квантованный (int8/ONNX)**. На этом CPU стартовать с
   **top-20–30 кандидатов, не 50**: кросс-энкодер по 50 парам может стоить 10+ секунд — юзер это
   видит на каждом вопросе. Замерить; если всё равно дорого — реранкать только при спорном fusion-скоре.
4. **Спойлер-фильтр** (payload-фильтр, применяется в ретриве — см. §6).
5. **Сборка контекста** — small-to-big: по найденным leaf тянем родителей по `parent_id`,
   **с дедупом по parent_id** (несколько хитов часто сидят в одной главе — иначе контекст
   задублируется). Бюджет входа генерации — **~4–6k токенов всего** (persona + вопрос + 2–3
   родителя): KV на этом железе дорогой, больше родителей = медленнее, а не лучше.
6. **Порог уверенности:** если лучший скор реранкера ниже порога — **честный отказ** («в
   проглоченной части такого нет»), персона обыграет. НЕ отвечать из общих знаний модели: для
   ридера галлюцинация про книгу хуже отказа. Порог подобрать на золотом наборе (§13, Фаза 1).
7. **Генерация ответа на 12B** с **цитатой места** (страница/глава для книги, кликабельный таймкод
   для видео). Верифицируемый RAG снимает вопрос «а не выдумал ли» и хорошо демается.

**Бюджет латентности и стриминг ответа (это ежедневный UX, в отличие от разового инжеста):**
- При single-digit decode ответ в 300 токенов — это **минута+ реального времени**. Убрать нельзя,
  спрятать можно:
- **Стримить ответ в Telegram** редактированием одного сообщения (`editMessageText`) с троттлингом
  ~раз в 1–2 с (флуд-лимиты Telegram), до первого куска — typing-индикатор. Ощущается живым, а не
  зависшим.
- **Кап на длину ответа** ~300–400 токенов; персона-стиль и так лаконичный.
- Пайплайн до генерации (эмбеддинг вопроса + ретрив + реранк) держать **≤ ~5 с**; главный кандидат
  на перерасход — реранк (шаг 3).

> Эмбеддер и реранкер — **мультиязычные** (bge-m3, bge-reranker-v2-m3). Контент русский, англоцентричные
> модели на нём просядут. Это не опция.

## 6. Спойлер-фильтр (одна из ключевых фич, гвоздь проекта)

Бот физически не может проспойлерить то, что читатель ещё не прочёл.

- Позиция читателя хранится **per (user, doc)** в SQLite.
- На ретриве добавляется payload-условие (пример под Qdrant):

```python
Filter(must=[
    FieldCondition(key="doc_id", match=MatchValue(value=doc_id)),
    FieldCondition(key="pos.offset", range=Range(lte=user_pos)),  # только то, что ДО позиции читателя
])
```

- **Сводки тоже фильтруются** этим же условием — потому что их `offset = max(children)` (§3).
- **Как узнать позицию читателя:**
  - команда `/pos` (глава/страница/процент);
  - **через vision:** юзер шлёт скриншот страницы, Gemma читает колонтитул с номером главы, бот сам
    двигает закладку. (Переиспользует существующий vision-модуль.)
- **Семантика позиции (зафиксировать, иначе фильтр будет тихо протекать):**
  - «читаю главу N» → разблокировано до **конца N−1**: текущая глава считается непрочитанной —
    безопасный дефолт, весь смысл фичи в гарантии отсутствия спойлеров;
  - «прочитал / закончил главу N» → до **конца N** (= max offset её leaf-чанков);
  - процент → `offset = pct × doc_len`; номера страниц издание-зависимы — конвертить только через
    главу/процент, иначе переспросить;
  - при неоднозначности бот задаёт **один** уточняющий вопрос; конвенция описана в help команды /pos.
- Персона-обвязка обыгрывает отказ: `Доступ к данным будущего закрыт. Продолжай чтение, мешок с костями.`

## 7. Тиры детализации: низкий / средний / высокий (+ ETA)

**Главная UX-фича.** Когда бот определяет контент как длинный, он предлагает выбрать глубину анализа
и **сразу показывает ETA** под замеренные tok/s. Это перекладывает wall-clock инжеста из проблемы
разработчика в осознанный выбор пользователя. На данном железе разброс между тирами — это **минуты
против ночи**, поэтому фича здесь не «приятно», а фактически обязательна.

| Тир | Что делает | Вызовов генерации LLM | Порядок времени (12B напрямую) |
|---|---|---|---|
| **Низкий** | Чистый retrieval, только эмбеддинг leaf-чанков. Все локальные вопросы работают. | **0** | минуты |
| **Средний** (дефолт) | + сводки на главу (для глобальных вопросов) | ~20–40 | десятки минут — ~1–2 ч |
| **Высокий** | + **Contextual Retrieval** (блёрб-контекст на КАЖДЫЙ чанк, приём Anthropic) — сильно бустит ретрив | ~600 | часы / ночь |

- Низкий тир может **вообще не грузить реранкер** — тир рулит не только качеством, но и тем, какие
  модели держать в памяти.
- Высокий тир гоняется **фоном**, бот пингует по готовности.
- **Инжест кэшируется на диск** — он одноразовый: стор + сводки в Qdrant/SQLite, переоткрытие
  документа = ноль пересчёта.
- **Тиры аддитивны** (низкий ⊂ средний ⊂ высокий): апгрейд = дозапуск только недостающих стадий,
  эмбеддинги не пересчитываются. Юзер может начать с низкого «прямо сейчас» и заказать высокий на ночь.

**Формула ETA:**

```
ETA ≈ (N_gen_calls × avg_output_tokens) / decode_tok_s   +   prefill_time
```

где `decode_tok_s` — **замеренный** на этом железе (llama.cpp его печатает), `avg_output_tokens` —
потолок выхода на вызов (см. ниже), `prefill_time` мал относительно decode. Пример для среднего:
~30 вызовов × ~400 out-токенов ≈ 12k генерённых токенов; при 20 tok/s ≈ 10 мин, при single-digit
на гибриде — соответственно дольше. **Реальный `decode_tok_s` нужно замерить — см. §12.**

**Рычаги ужать время (важно, часть контринтуитивна):**
- **Короткий output, НЕ input.** Скормить главу (input) дёшево — это prefill; дорого — генерить
  длинную сводку. Ставить потолок ~300 токенов на выход.
- **Prefix caching** для contextual-тира: общий контекст главы шарится между её чанками, KV-кэш
  переиспользуется → prefill почти бесплатный.
- **Grind-модель** (§8) — опциональная ускорялка высокого тира.

## 8. Опциональная grind-модель (ускорялка инжеста)

Суммаризация/контекстуализация на инжесте — это **пакетная тупая работа** «ужми главу / опиши
чанк». Качество 12B тут не нужно.

- Поднять на **инжест-грайнд отдельную маленькую модель** (2–4B, Q4). Она влезает в 4 ГБ VRAM
  **целиком** → full-GPU без оффлоада → ~30–50 tok/s (а не single-digit гибрида).
- Тогда высокий тир из «ночи» → ~1–2 ч, средний → ~10–20 мин.
- 12B бережётся для **user-facing ответов**, где качество реально решает.
- Это **вторые веса** в памяти (в отличие от контекстов на §1), но в Q4 маленькая модель рядом с
  12B по RAM влезает.

**Статус:** опционально. Благодаря тирам это не обязательно — юзер может выбрать высокий на 12B
фоном, осознанно приняв время. Grind-модель = «хочу высокий за 20 мин вместо ночи».

## 9. Раскладка моделей и памяти (специфично для 4/16)

**VRAM (4 ГБ):**
- GPU — основная 12B (partial offload). На инжест-фазе с grind-моделью GPU может отдаваться под
  маленькую модель (она влезает целиком).
- **Эмбеддер (bge-m3) и реранкер — на CPU.** На запросе эмбеддится один короткий вопрос (CPU —
  мгновенно); реранкер — int8, топ-20–30 пар, бюджет по времени см. §5 шаг 3. VRAM целиком под LLM.

**RAM (16 ГБ) — впритык, бюджетировать:**
```
OS ~3.5  +  часть 12B в RAM ~4  +  bge-m3 ~2.5  +  реранкер ~2.5  +  оверхед  ≈  ~14 ГБ
```
Спасение — не держать всё одновременно, а разводить по фазам:
- **инжест-фаза:** (grind-модель или 12B) + эмбеддер;
- **чат-фаза:** 12B + эмбеддер + реранкер.

## 10. Хранилища

- **Векторный стор — Qdrant.** Killer-фича под этот проект — **payload-фильтры** (основа
  спойлер-фильтра, §6). Хранит named vectors (dense+sparse), фьюзит RRF на своей стороне.
  - *На вырост (не сейчас):* если корпус когда-нибудь дорастёт до **библиотеки** (десятки-сотни книг
    + видео, сотни тысяч / миллионы векторов), Qdrant 1.18 умеет TurboQuant-квантизацию как **флаг в
    конфиге + реиндекс**, фильтрация при этом сохраняется. На масштабе одной книги стор — единицы
    МБ, компрессия не нужна. Упоминается только чтобы не закладываться на неё раньше времени.
  - Альтернатива, если не хочется отдельного сервера: LanceDB (embedded). Но payload-фильтры Qdrant
    под спойлеры ложатся чище.
- **Стейт бота — SQLite:** юзеры, документы, **позиции чтения per (user, doc)**, статусы фоновых
  джоб.

## 11. Гочи и подводные камни (специфичные для этого железа/контента)

- **Continuous batching почти не поможет.** Гибридный инференс упирается в bandwidth RAM, а не в
  compute; параллельные слоты грызут одну полосу → прирост скромный. Не закладываться на него как на
  ускоритель инжеста на этой машине.
- **Главы режем скромно.** KV-кэш на 4 ГБ дорогой — не кормить 30k-токенные мега-главы одним куском.
- **Эмбеддер/реранкер строго мультиязычные** (bge-m3 / multilingual-e5, bge-reranker-v2-m3). Русский
  контент, англоцентричные модели просядут.
- **Парсинг форматов — отдельный ад:** FB2/EPUB/PDF с колонками, сносками, сканами под OCR. Обычно
  съедает больше времени, чем сам RAG. Заложить на это реальный бюджет.
- **Инжест длинного видео стримить с прогрессом** — 40 минут тишины = «завис».
- **Эмбеддинг ≠ генерация.** Основная масса инжеста (парсинг, чанкинг, эмбеддинг) — дёшево; дорогой
  только слой генерации сводок/контекста. Не путать при оценке времени.
- **Инжест и живой чат конкурируют за одни веса и одну полосу RAM.** Не гонять сводку параллельно с
  ответом юзеру — обоим будет вдвое хуже. App-level приоритет: инжест-воркер проверяет флаг
  «активный запрос» **между** generation-вызовами и уступает; сводка ≤300 токенов, так что юзер
  ждёт максимум один вызов.
- **KV-кэш можно квантовать** (флаги llama.cpp: cache-type-k/v = q8_0; для V может требоваться
  включённый flash-attention). На 4 ГБ это заметно освобождает память под контекст ценой небольшой
  потери качества — замерить на своих промптах.

## 13. Предлагаемый план по фазам

- **Фаза 0 — измерение.** Замерить `decode_tok_s` и `prefill` на 12B Q4 на целевой машине; заодно —
  нативный аудио-тракт (§12, п.5). Прошить в ETA-формулу и в порог fast-path.
- **Фаза 1 — MVP лонг-ридера (низкий+средний тир, только книги).** Парсинг EPUB/FB2 → дерево →
  позиционные якоря → эмбеддинг (bge-m3, CPU) → Qdrant с payload → гибрид+реранк → small-to-big →
  генерация с цитатой. Сводки глав для среднего тира. Спойлер-фильтр. SQLite-стейт. Инжест как
  фоновая джоба с прогрессом. Роутер локальный/глобальный. Стриминг ответа в Telegram.
  **Acceptance: золотой набор** — 10–20 вопросов по одной тестовой книге с известной главой-источником,
  метрика hit@k (попал ли нужный источник в топ). Дёшево, ловит регрессии при кручении ретрива и
  даёт порог для шага «честный отказ» (§5, шаг 6).
- **Фаза 2 — высокий тир + тир-UX.** Contextual Retrieval, выбор тира с ETA, (опц.) grind-модель.
- **Фаза 3 — аудио/видео.** Короткое — fast-path нативно, без инжеста. Длинное: приём ссылкой →
  yt-dlp (audio-only) → транскрипт с таймкодами (бэкенд по замерам §12: faster-whisper или нативно
  чанками) → RAG. Цитаты как кликабельные таймкоды.
- **Фаза 4 — мультимодалка видео (опц.).** PySceneDetect → vision → отдельный визуальный индекс.
- **Фаза 5 — PDF/OCR** и прочие грязные форматы.

## Приложение: сводка ключевых решений (быстрая справка)

| Решение | Выбор | Почему |
|---|---|---|
| Стратегия контекста | RAG, не длинный контекст | 1M токенов не влезает; даже KV-компрессия 6× от 1M — гигантомания на 4 ГБ |
| Чанкинг | дерево + позиционные якоря | основа small-to-big и спойлер-фильтра |
| Что эмбеддить | leaf-чанки, тянуть родителя | острый поиск + достаточный контекст, 0 LLM-вызовов |
| Эмбеддер | bge-m3 (гибрид dense+sparse), CPU | sparse ловит имена собственные; мультиязычность под русский; VRAM под LLM |
| Реранкер | bge-reranker-v2-m3, CPU | буст качества; на CPU не жрёт VRAM |
| Стор | Qdrant | payload-фильтры под спойлер-фильтр |
| Стейт | SQLite | юзеры, позиции чтения, статусы джоб |
| Глобальные вопросы | сводки на главу (1 уровень) | ~30 вызовов, не N² (полный RAPTOR — оверкилл) |
| Тиры | низкий(0)/средний(~30)/высокий(~600) вызовов + ETA | wall-clock → выбор юзера |
| Grind | опц. маленькая модель на инжест | 2–4B влезает в 4 ГБ целиком → ~30–50 tok/s |
| Continuous batching | НЕ закладываться | гибрид упирается в bandwidth RAM |
| Единица offset | символы plain text (книги) / секунды (видео) | закладки не ломаются при смене токенизатора |
| Видео в бота | ссылкой → yt-dlp, audio-only | Bot API не отдаёт файлы >20 МБ |
| Ответ юзеру | стриминг editMessageText, кап ~300–400 ток. | single-digit decode = минута на ответ |
| Ретрив-мисс | порог скора реранкера → честный отказ | галлюцинация про книгу хуже отказа |
| Дубли документов | doc_id = sha256 файла | повторная заливка ≠ повторный инжест |
| Короткое аудио/видео | нативно, без инжеста | мультимодальность = fast-path; порог по замеру токен-рейта |
| Транскрипция длинного | faster-whisper (CPU) по умолчанию, нативно чанками как опция | LLM-транскрипция decode-bound ≈ реальное время+; whisper не занимает LLM |
